# Phase G-1 — Trigger configuration UI + fire log

## Summary

Built the human-facing UI for attaching triggers to active strategies, plus
the per-trigger fire log view. This is the centerpiece of Phase G —
Phase F shipped the backend (entities, evaluator, agent invocation, fire
log API, kill switches); G-1 surfaces it for humans, agents-only via API
isn't enough.

Branch: `feat/g1-trigger-ui` (forks `main` at `5a36c67`).

## What landed

### New page: Activation detail

Route: `/activations/:id`. Reached by clicking a row in the activations
list. Renders:

- Hero: eyebrow + strategy name + status badge + creation/last-run captions.
- Status panel: target portfolio, frequency, strategy. Surfaces
  `last_error` on ERROR-status activations.
- Triggers section (the centerpiece — see below).

### New section: TriggersSection

Embedded on the activation detail page. Shows the activation's triggers
as a hairline `<DataTable>`:

- Status badge (`bg-gain-soft` / `text-ink-subtle` / `text-ink-faint` /
  `bg-loss-soft` for ACTIVE / PAUSED / EXPIRED / MANUALLY_DISABLED).
- Condition summary (e.g. "Drawdown > 5% over 3d (portfolio)").
- Cooldown (formatted: 21600s → "6h").
- Truncated agent prompt (hover to reveal full).
- Last fired (relative time, "—" if never).
- Per-row actions: View fires, Pause/Resume, Edit, Delete.

Empty state uses the editorial `<EmptyState>` pattern with an explicit
"Attach trigger" CTA.

Per Phase F design Q3: MANUALLY_DISABLED triggers show a "Recreate" CTA
instead of pause/edit/delete (PATCH would 422 on a terminal-state row).

### New dialog: CreateTriggerDialog

Modal with a dynamic params section driven by the condition-type select.
Contracts:

- DRAWDOWN_THRESHOLD: `threshold_pct` (0–100), `lookback_days` (1–365),
  `metric` radio (Portfolio / Per ticker).
- VOLATILITY_SPIKE: `threshold_pct` (0–500), `over_days` (5–90).
- EARNINGS_PROXIMITY: `days_before` (1–14).
- CUSTOM_RULE: rendered as disabled option per Phase F Q1.

Cooldown is composed from a number + unit (minutes/hours/days) and
translated to seconds at submit time.

Default-API-key picker only renders when the user has more than one
trade-scoped, non-revoked key — with zero or one, the backend fallback
rule ("most-recently-used trade-scoped key") handles attribution
implicitly.

Validation is client-side; backend is the source of truth (it'll 422 on
malformed params).

### New dialog: EditTriggerDialog

Smaller inline edit modal. Restricts to fields that change shape during
day-to-day tuning: cooldown + agent_prompt. Condition type and params are
intentionally out of scope (heavy enough that "delete and recreate" is
the cleaner UX). Status changes go through one-click pause/resume buttons,
not the edit dialog.

Initial values are seeded via the `key` prop pattern (so each dialog
open re-mounts with fresh state) — no `useEffect` to sync props to state.

### New page: TriggerFireLog

Route: `/triggers/:id/fires`. Reached from the "View fires" action on
the activation detail page. Renders:

- Hero: condition summary headline + status badge.
- Caption: cooldown, last_fired_at, created_at.
- Operator-instruction panel (the trigger's `agent_prompt` verbatim).
- Fires table with columns: Time, Decision (badge), Snapshot, Rationale
  (truncated), Trade (link or task link), Latency.

Decision badge palette per Phase F design:

- BUY → `text-gain`
- SELL → `text-loss`
- HOLD → `text-ink-subtle`
- MODIFY → `text-amber`
- NEEDS_HUMAN → `bg-amber-soft` pill
- INVOCATION_FAILED → `bg-loss-soft` pill

Empty state explains the trigger is active but its condition hasn't been
met.

### Hooks + service

- `frontend/src/services/api/triggers.ts` — wraps `POST/GET/PATCH/DELETE
  /triggers/...` and `GET /activations/:id/triggers` + `GET
  /triggers/:id/fires`. Matches the patterns in `strategies.ts` /
  `apiKeys.ts`.
- `frontend/src/hooks/useTriggers.ts` — TanStack Query hooks:
  `useTriggers(activationId)`, `useTrigger(id)`, `useTriggerFires(id)`,
  `useCreateTrigger`, `useUpdateTrigger`, `useDeleteTrigger`. Mutations
  invalidate the list and per-id caches on success.
- `frontend/src/hooks/useStrategyActivation.ts` — added
  `useActivationById(id)` (filters the list endpoint client-side; the
  backend has no `GET /activations/:id` yet — left as a follow-up
  noted in the PR body).

### Formatters

- `formatCooldown(seconds)` — coarsening units (300 → "5m", 21600 → "6h",
  90000 → "1d 1h").
- `formatConditionSummary(type, params)` — discriminated, falls back to
  the bare condition type when shape is unrecognised.
- `formatFireSnapshot(fire)` — extracts the most useful single fact from
  `condition_evaluation_data` (drawdown_pct, realised_vol_pct, days_until)
  with a key-list fallback.

## Backend-contract notes

A few API/UI naming nuances surfaced; calling them out in the PR body so
the next maintainer doesn't have to re-derive them:

- The backend uses **British "realised"** in the
  `condition_evaluation_data` for VOLATILITY_SPIKE
  (`realised_vol_pct`). The fire-log formatter accepts both spellings as
  a defensive fallback.
- `condition_params.threshold_pct` arrives on the wire as a **string**
  (Decimal precision); UI inputs render and submit it as a string.
  `lookback_days` / `over_days` / `days_before` are integers.
- The `default_api_key_id` field is **optional** on create — when null,
  the backend's "most-recently-used trade-scoped key" fallback applies.
  The UI hides the picker when the user has zero/one trade-scoped key
  to keep the form quiet.

## Out of scope for G-1 (deferred)

Per the task spec:

- Strategy provenance display (who created the trigger) — G-2.
- Agent activity drill-down per fire — G-2.
- "Ask an agent" button — G-2.
- Kill-switch UI — separate small follow-up; backend
  (`POST /triggers/disable-all` and `POST /admin/triggers/disable-all`)
  is shipped, just not surfaced in the UI yet.

A backend follow-up worth noting:
`GET /activations/:id` doesn't exist. The detail page filters the list
endpoint client-side; for users with > 100 activations this would miss.
Acceptable trade-off for G-1 (most users have < 5 activations).
