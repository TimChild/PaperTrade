# Phase C1.2 + C1.3 — Live Strategy Execution Service, Scheduler Integration, API Endpoints

**Branch**: `feat/phase-c1.2-c1.3-live-execution`
**Task**: `agent_docs/tasks/210_live_strategy_execution.md` Phases 2 + 3
**Predecessors**: PR #235 (StrategyActivation entity), PR #237 (API-key auth)

## What landed

### Service (`zebu.application.services.strategy_execution_service`)

`StrategyExecutionService` mirrors `BacktestExecutor`'s shape for a single
calendar day:

- `execute_active_strategies()` — scheduler entry point. Lists every
  `ACTIVE` activation, dispatches each one through `_safe_execute`, returns
  a `{processed, succeeded, failed, trades}` summary.
- `execute_one(activation_id)` — manual `run-now` entry point. Returns a
  per-activation result envelope.
- Per-activation isolation: any exception thrown while running one
  activation is captured on that activation's record (status flips to
  `ERROR`, `last_error` set). The cycle continues.
- The signal-to-transaction translation reuses `BacktestTransactionBuilder`
  (also used by `BacktestExecutor`) so live and backtest mode share the
  same trade-factory invariants. A new `seed_holdings()` helper on the
  builder loads the live portfolio's open positions before applying SELL
  signals.
- Strategy-type routing duplicates the small `match` block from
  `BacktestExecutor._build_strategy` rather than depending on it (live
  and backtest are sibling services; lifting this is a refactor candidate
  if a third call site appears).

### Scheduler (`zebu.infrastructure.scheduler`)

New `execute_active_strategies` job + `SchedulerConfig` extension
(`strategy_execution_cron`, `strategy_execution_enabled`). Default cron is
`30 0 * * 1-5` (00:30 UTC weekdays — sits 30 min after the existing
price-refresh job at 00:00 UTC, weekdays only because `DAILY_MARKET_CLOSE`
makes no sense on closed-market days). `replace_existing=True` keeps the
registration idempotent.

### API endpoints (`zebu.adapters.inbound.api.strategy_activations`)

Five routes across two routers:

| Route | Verb | Auth |
|---|---|---|
| `/strategies/{id}/activate` | POST | Bearer JWT |
| `/strategies/{id}/activation` | GET | Bearer JWT |
| `/activations` | GET | Bearer JWT |
| `/activations/{id}/deactivate` | POST | Bearer JWT |
| `/activations/{id}/run-now` | POST | **Bearer JWT or API key** (Phase C3) |

`run-now` accepts both auth schemes via the unified `CurrentUserDep` from
PR #237. Ownership checks compare `activation.user_id`/`strategy.user_id`/
`portfolio.user_id` against the deterministic UUID derived from the
authenticated identity — same UUID regardless of which scheme the request
came in on.

## Tests

- **Service unit tests (13)** — empty active list, successful run, one
  failing activation doesn't block others, paused activations skipped,
  missing portfolio → ERROR, zero signals → no-op, missing price doesn't
  fail, run-now on missing id, run-now on paused activation, DCA routing,
  MA Crossover routing, holdings seeding, SELL-against-existing-holdings.
- **Integration tests (25)** — activate happy path, invalid frequency,
  missing strategy/portfolio, double-activate → 409, ownership 403s,
  get/list/deactivate/run-now happy paths, list pagination, list scoping,
  unauthenticated rejection, **run-now via X-API-Key auth (C3)**.
- **Scheduler tests (4 new)** — config defaults including new fields,
  job registration idempotency, disabled-by-config, `execute_active_strategies`
  is a coroutine fn.

Coverage on new code: 100% on service, 99% on adapter. Total backend
tests: 1303 (was 1298 — net +42 added: 38 in new files + 4 new
scheduler tests).

## Smoke test

`test_strategy_execution_job_registered_by_default` boots the scheduler
end-to-end and asserts the job appears in `scheduler.get_jobs()` with the
expected ID. That's the manual smoke the spec asked for, in test form.
